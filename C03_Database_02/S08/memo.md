# S08. 인덱스 1

## 인덱스가 필요한 이유

### 문제 상황
```sql
select * from items where item_name = '게이밍 노트북';
```
- 데이터가 몇 건 없을 때는 눈 깜짝할 사이에 끝나는 이 쿼리가
- 데이터가 수백만건이 쌓이면 점점 느려짐

### 느린 검색의 원인: 풀 테이블 스캔(Full Table Scan)
- 인덱스가 없는 테이블에서 특정 데이터를 찾는 것은
- 책에서 특정 단어를 찾기 위해서 첫페이지부터 마지막페이지까지 전부 다 넘겨보는 것과 같음
- 어디쯤에 있는지 "힌트"가 필요함 => 이것이 인덱스!!
    - 최악의 경우: 마지막 행에 있거나, 없으면 모든 행을 다 찾아봐야만 함
    - **항상 최악의 경우를 생각해야만 함** - O(N)

### 데이터 양에 따른 성능 저하
- 정확한 응답 시간을 예측할 수 없음
- 데이터베이스 서버의 사양, 환경 설정, 한 행의 크기, 시스템의 현재 부하 등 수많은 변수에 따라 크게 달라짐

### 실무 이야기
- 풀테이블 스캔은 매우 비용이 높은 작업
- 가급적 풀테이블 스캔이 발생하지 않도록 설계하는 것이 필수
- 해결 방법
    - **인덱스 활용**: where 절에 자주 사용되는 검색 조건 컬럼에는 인덱스를 생성하여 풀테이블 스캔을 방지
    - **실행 계획 확인**: 쿼리 실행 전 실행 계획(Execution Plan)을 확인하여 의도치 않은 풀테이블 스캔이 발생하는지 반드시 점검해야함
    - **작업 시간 분리**: 대용량 데이터에 대한 전체 스캔이 불가피한 통계/배치 작업이라면, 서비스 이용자가 적은 새벽 시간에 실행하는 것을 권장함

## 인덱스 소개

### 비유
- 책의 비유에서, 특정 단어를 찾을 때, 책의 맨 뒤에 있는 찾아보기(색인) 페이지를 활용함
- 찾아보기 페이지
    - 알파벳순, 가나다순(정렬된 순서)로 정리되어있음
    - 각 키워드 옆에는 그 키워드가 등장하는 페이지 번호가 적혀 있음
- 중요한 개념
    - **정렬된 순서로 정리되어있음**

### 개념
- 특정 컬럼들의 데이터를 기반으로 생성되는 원본 테이블과는 **별개의 특수한 자료 구조**
    - 지정된 컬럼의 값과 해당 값을 가진 실제 데이터 행의 위치(주소값, 포인터, PK 값 등)을 한 쌍으로 저장함
    - 가장 중요한 것은 **항상 정렬된 상태를 유지함**

### item_name 기반의 인덱스
- item_name | 원본 데이터 위치(PK: item_id)
- 가나 | 2
- 나다 | 11
- 다라 | 6
- 라마 | 20
- ...
- 포인트
    - item_name이 오름차순으로 정렬되어있음
    - 원본 데이터의 위치를 쌍으로 저장하고있음(항상 PK일 필요는 없음 - **인덱스 종류마다 다름**)

- 정렬된 데이터를 빨리 찾을 수 있음
    - 기본 아이디어: 이진 탐색(O(logN))

### 실제 인덱스의 구현(대략적)
- RDB에 종속된 복잡한 구현 세부사항을 다루지 않음
- 일반적이고 논리적인 인덱스 개념을 중심으로 설명
- 실제 물리적인 인덱스는 그 종류에 따라 구현이 다름

- MySQL
    - 클러스터 인덱스 / 보조 인덱스
    - 클러스터 인덱스: PK를 기반으로 만드는 인덱스
        - 원본 데이터 자체를 인덱스와 함께 보관해서 원본 데이터를 매우 빠르게 찾을 수 있음
    - 보조 인덱스
        - 원본 데이터의 기본 키(PK) 값을 함께 보관함
        - 이 기본 키 값으로 클러스터 인덱스를 통해 원하는 데이터를 조회

## 트리 자료 구조

### 개념
- 부모 노드와 자식 노드로 구성됨
    - 노드 안에는 (데이터, 다음 노드들의 위치)가 보관됨
- 가장 높은 조상을 루트 노드라고 함
- 자식이 2개까지 올 수 있는 트리를 이진 트리라고 함
- 노드의 왼쪽 값은 더 작은 값을 가지고, 오른쪽 값은 더 큰 값을 가지는 것을 **이진 탐색 트리**

### 이진 탐색 트리
- 핵심: 데이터를 입력하는 시점에 데이터를 정렬해서 저장함

### 이진 탐색 트리 - 검색
- 부모보다 작음 => 왼쪽으로 이동
- 부모보다 큼 => 오른쪽으로 이동
- 시간 복잡도: O(logN)

### 이진 탐색 트리 - 순회
- 정렬해서 보관을 하기 때문에 정렬된 순서로 데이터를 차례로 조회할 수 있음(순회)
- 데이터를 차례로 순회하려면 중위 순회라는 방법을 사용하면 됨
    - 왼쪽 서브트리를 방문
    - 현재 노드 처리
    - 오른쪽 서브트리 방문
    - **오름차순**

### 이진 탐색 트리 문제점
- 데이터 저장 시, 오름차순으로 저장한다면?
- 오른쪽으로만 치우져지는 트리가 생성됨
    - 검색 속도(O(N))

### 이진 탐색 트리 개선
- 트리의 균형을 유지하는 것은 밸런스 트리라고 함
- 데이터베이스는 대부분 인덱스에 밸런스 트리를 사용해서 균형을 유지

### 실제 인덱스에 사용되는 트리
- (값, 원본 데이터 위치) 쌍으로 저장함
- 실제로는 B-Tree 구조!! (B-Tree / B+-Tree)
- B트리는 밸런스 트리의 한 종류
- 단순히 균형을 유지하는 것을 넘어서, 대용량 데이터를 다루는 한경에 최적화되어있음
- 디스크 I/O 횟수를 줄이는 것이 성능에 결정적인 영향을 미침
- 이진 탐색 트리는 한 노드가 하나의 데이터만 가지고 있어, 데이터를 검색할 때 여러 노드를 방문해야함
- 그만큼 디스크에서 많은 블록을 읽어야함
- 한 번 데이터를 읽을 때 더 많은 정보를 가져올 수 있게 하여 디스크 I/O 횟수를 획기적으로 줄여줌

## 인덱스 생성, 조회, 삭제

### CREATE INDEX: 인덱스 생성하기
```sql
create index 인덱스이름 on 테이블이름 (컬럼1, 컬럼2, ...);
```
- 인덱스 이름: 생성할 인덱스에 고유한 이름을 붙여줌: idx_테이블명_컬럼명 규칙을 보통 지음
- 테이블 이름: 인덱스를 생성할 대상 테이블
- 컬럼: 인덱스를 구성할 컬럼: 하나 이상의 컬럼을 지정할 수 있음

### SHOW INDEX: 인덱스 조회
```sql
show index from 테이블이름;
```
- MySQL에서는 기본키, 외래키 제약조건을 설정하면 자동으로 인덱스를 생성함
- UNIQUE 제약조건도 인덱스가 생성됨
- Non_unique
    - 1이면 중복 값을 허용하는 인덱스, 0이면 중복을 허용하지 않는 고유 인덱스(UNIQUE, PRIMARY KEY)
- Cardinality: 인덱스에 저장된 유니크한 값의 개수에 대한 추정치.
    - 이 값이 높을수록 중복도가 낮다는 의미이며, 인덱스의 성능이 좋다고 판단할 수 있음

> UNIQUE 제약 조건에 인덱스를 자동으로 생성하는 이유
- 새로운 데이터 삽입 시, 현제 데이터베이스에 유니크 조건을 위배하지 않는지 검색하기 위해서는 테이블을 찾아야함. 인덱스가 없으면 풀 테이블 스캔

### DROP INDEX: 인덱스 삭제하기
```sql
drop index idx_item_name on items; 
```

### 인덱스가 정말 사용되는지 확인하는 방법(EXPLAIN)
- 데이터베이스에는 쿼리를 어떤 방식으로 최적화해서 실행할지 계획하는 기능이 존재
- "쿼리 옵티마이저"
- **인덱스를 만들었다고해서 select문에 항상 그 인덱스를 사용하는 것이 아님**
- 풀 테이블 스캔이 더 빠른 경우도 있음

### 실행 계획
```sql
explain select * from items where item_name = '게이밍 노트북';
```
- id
- type
    - ALL: 가장 중요하게 봐야할 부분, ALL은 풀테이블 스캔을 의미함
    - 인덱스를 제대로 사용했다면, ref, range 등 다른 값이 표시됨
        - ref는 = 조건이나 join에서 인덱스를 사용했다는 의미
        - range는 범위 검색(between, >, <, >=등)에서 인덱스를 사용했다는 의미
- key
    - null: 어떤 인텍스도 사용하지 못했다는 의미
- rows
    - 쿼리를 처리하기 위해서 탐색할 것으로 예측하는 행의 수
    - 실제 실행하는게 아니라 데이터베이스 나름의 통계 데이터를 기반으로 하는 **예측 정보**
    - 정확하게 맞지 않음
- filtered
    - 10.00: 테이블에서 읽어온 행들 중에서 where 조건으로 필터링되고 난 후, 최종적으로 남을 것으로 행의 수(필터링을 통과할 것을 예상되는 행의 수)
- extra
    - using where: 데이터를 가져온 후에 where절의 조건을 사용해 필터링 작업을 수행했다는 의미
        - 인덱스를 효율적으로 사용했다면, 처음부터 조건에 맞는 데이터만 골라서 가져왔을것임
        - 인덱스가 없으민 모든 데이터를 가져와서, 그 후에 조건에 맞는지 일일이 비교하는 비효율적인 방식

## 인덱스와 동등 비교

### 인덱스를 사용하는 상황
- 동등 비교(=)
- 범위 검색(between, >, <, like 등)
- order by를 통한 정렬 작업

### 인덱스와 동등 비교 시작
- type: ref: 동등 비교 조건이나 join에서 인덱스를 사용했다는 의미
- possible_keys: 현재 쿼리에서 사용 가능한 인덱스의 후보군
    - 지금은 하나밖에 없음
- key: idx_items_item_name: 사용할 인덱스 이름
- filtered: 100.00: 인덱스를 통해서 찾은 1개의 행을 100% 선택한다는 뜻
- rows: 1: 인덱스가 없을 때는 테이블 전체 행을 스캔할 것으로 예측했지만
    - 이제는 단 1개의 행만 스캔할 것이다라는 의미
- extra: null: 데이터를 가져온 후에 별도의 필터링 작업이 필요가 없다는 의미

> 중요: 인덱스 강제로 사용하기
- 샘플 데이터가 적으면 데이터베이스 환경에 따라 인덱스를 활용하지 않는 실행 계획을 짤 수 있음
- 강제로 적용
```sql
explain select * from items force index(idx_items_item_name)
where item_name = '게이밍 노트북';
```

## 인덱스와 범위 검색

### 인덱스와 범위 검색 시작
- 인덱스가 없을 때
```sql
explain select * from items where price between 50000 and 100000;
-- 풀 테이블 스캔
```

- 인덱스 생성
```sql
create index idx_items_price on items(price);
```

```sql
explain select * from items where price between 50000 and 100000;
-- type: range: 범위 검색
```
- 동작 원리
    - 50000원 찾고
    - 인덱스에서 다음행, 다음행, ...
    - 100000을 넘는 순간 탐색 종료 (그 다음부터는 다 넘을 것이기 때문)

### 조금 이상한 점
- 결과의 순서가 바뀌었음
- 인덱스가 없을 때는 id 순으로 출력됨
- 인덱스가 있으니깐 가격 순으로 출력됨

- 인덱스가 정렬(가격순)되어있고, 인덱스를 기준으로 찾기 때문에 가격 순으로 되어있음
- **데이터베이스가 항상 보장하는 것은 아님**
- **정렬이 필요하면 무조건 order by price로 하는 것을 권장함**

## 인덱스와 Like 범위 검색

### Like절에서 인덱스를 사용하기 위해서 주의사항
- 와일드카드가 검색어의 뒤쪽에 위치해야만 함!!
    - where item_name like '게이밍%': 가능
    - where item_name like '%게이밍': 불가능
    - where item_name like '%게이밍%': 불가능
- %가 앞에 있으면 시작 시점이 불분명해져서 인덱스를 사용할 수 없음!!

### 예제
```sql
explain select * from items where item_name like '게이밍%';
```
- 범위 검색을 사용함
    - 글자가 정렬되어있기 때문
    - 게이밍 노트북 / 게이밍 의자 / .. 넘어가면 없음(검색 종료)

### 원리
- 게이밍으로 시작하는 조건을 찾음
- item_name이 정렬되어있기 때문에 게이밍 노트북, 게이밍 의자, SQL 마스터 가이드(탐색 종료)
- **게이밍으로 시작하는 가장 첫 단어를 먼저 찾으면 됨**

### LIKE 검색 실패 예제: 와일드 카드 앞에 올때
```sql
explain select * from items where item_name like '%게이밍%';
-- 인덱스를 탈 수 없음
```

### 실무 팁: 전문 검색, 전체 문자 검색(Full-Text Search)
- like의 %검색어%는 성능이 심각하게 저하되어 실제 서비스에서는 사용하기 어려움
- 데이터베이스는 전문 검색이라는 특수한 기능을 제공함
- 전문 검색 인덱스는 B-Tree와 달리, 텍스트를 단어 단위로 쪼개서 인덱싱하는 방식
- 이를 통해 텍스트 중간에 있는 단어도 매우 빠르게 검색할 수 있음
- **match ... against** 구문을 사용하는 전문 검색 기능을 도입하는 것이 해결 방법
- **ElasticSearch**

## 인덱스와 정렬

### 인덱스를 활용한 정렬
- order by가 인덱스를 잘 활용하면, 이미 정렬된 인덱스를 순서대로 읽기만 하면 되므로, 매우 빠르게 동작함
```sql
explain select * from items order by stock_quantity;
```
- 이렇게 하면 데이터베이스가 stock_quantity를 가지고 Using filesort를 통해서 정렬을 다시 해줌

### 인덱스를 사용해 정렬까지 한 번에 처리하는 경우
- 가장 이상적인 상황은 where절의 조건과 order by절의 정렬 기준이 같아서, 인덱스 하나로 검색과 정렬을 모두 해결하는 경우
- price 정렬
```sql
explain select * from items where price between 50000 and 100000
order by price;
```
- **using index condition**
- 이미 인덱스로 정렬이 되어있음
- 인덱스를 순서대로 찾으면 이미 정렬된 순서임
- 따라서 별도의 filesort 작업이 필요없음

### 역방향으로 조회하는 경우
- 단일 컬러 인덱스에서는 filesort없이 효율적인 처리가 가능함
- 그냥 역방향 스캔을 하면 됨
```sql
explain select * from items where price between 50000 and 100000
order by price desc;
```
- Extra: Using index condition; **Backward index scan**

> 정방향 스캔이 역방향 스캔보다 미세하게 더 빠름 - 미미해서 큰 차이 없음

### 내림차순 인덱스(Descending Index)
- 역방향 스캔은 효율적이지만 정렬 방향과 일치하는 인덱스를 직접 만들 수도 있음
- MySQL 8.0부터는 내림차순 인덱스를 지원함
```sql
drop index idx_items_price on items;
create index idx_items_price on items (price DESC);

explain select * from items where price between 50000 and 100000
order by price desc;
--- Extra: Using index condition
```
