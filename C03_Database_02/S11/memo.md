# S11. 트랜잭션

## 트랜잭션이 필요한 이유

### 데이터베이스 일관성을 위협하는 요소
- 상태
- **행위**: 주문하기
    - 작업1: orders 테이블에 새로운 주문 정보를 insert함
    - 작업2: products 테이블에서 방금 주문된 상품 재고를 update하여 1만큼 줄임(정확히는 주문 수만큼)
    - **이 두 작업은 논리적으로 쪼개지면 안되는 하나의 묶음**

### 트랜잭션이 없다면?
- 시나리오
    - 1번 작업 성공 / 2번 작업 실패
    - 주문 테이블은 성공 / 재고 감소 실패
    - 일관성이 깨짐
        - 주문이 들어갔는데도 재고가 있다고 착각할 수 있음

- 문제
    - 재고 관리의 실패
    - 고객 신뢰도 하락
    - 잘못된 데이터 분석
    - 반대도 동일함

### 해결책: 트랜잭션
- 논리적으로 절대 쪼갤수 없는 하나 이상의 데이터베이스 작업 묶음이라는 원칙
- 묶인 작업들은 모두 다 성공해야만 영구적으로 반영
- 작업 내 단 하나의 작업이라도 실패하면, 그전에 성공했던 모든 작업들을 전부 없었던 일로 되돌려 버림

### 주문 처리 시나리오를 트랜잭션으로 묶었을 때
- 주문 처리 트랜잭션 시작
- insert문으로 orders 테이블에 주문을 기록 (성공)
- update문으로 products 테이블의 재고를 줄임 (실패)
- 하나의 작업이라도 실패했으므로, 트랜잭션은 실패로 간주
- 데이터베이스는 2번에서 성공했던 insert 작업까지 자동으로 취소(원상 복구)시킴
- 결과적으로 orders, products은 이 트랜잭션이 시작되기 전에 완벽하게 일관된 상태로 남게 됨
- **All or Nothing**

## 커밋, 롤백

### 핵심 명령어 3가지
- START TRANSACTION: 지금부터 트랜잭션을 시작해라
    - 이 명령어 이후에 실행되는 모든 쿼리는 하나의 트랜잭션으로 묶여 관리됨(BEGIN)
- COMMIT: 트랜잭션 내의 모든 작업이 성공적으로 완료되었으니, 지금까지 변경 사항을 디스크에 영구적으로 저장해라
    - COMMIT이 실행된 후에는 변경 내용을 되돌릴 수 없음
- ROLLBACK: 문제가 발생하여 작업을 중단해라
    - 이 트랜잭션 내에서 실행했던 모든 변경 사항을 전부 취소하고, 트랜잭션이 시작되기 직전의 상태로 되돌림

### 실습: 계좌이체 시나리오
```sql
DROP TABLE IF EXISTS accounts;
CREATE TABLE accounts (
 id INT PRIMARY KEY,
 owner_name VARCHAR(255) NOT NULL,
 balance INT NOT NULL
);
-- 초기 데이터 입력
INSERT INTO accounts (id, owner_name, balance) VALUES
(1, 'A고객', 50000),
(2, 'B고객', 20000);
```

```sql
start transaction;

update accounts set balance = balance - 10000 where id = 1;
select * from accounts;

update accounts set balance = balance + 10000 where id = 2;
select * from accounts;

-- 여기까지는 임시!!

commit; -- 영구저장
select * from accounts;
```

```sql
start transaction;

update accounts set balance = balance - 5000 where id = 1;
select * from accounts;

-- 여기까지 임시!!

rollback;

select * from accounts; -- 원상복구
```

### MySQL의 autocommit
- MySQL은 기본적으로 autocommit 모드가 활성화(ON, 1)되어있음
- 반드시 start transaction 명령어로 명시적으로 트랜잭션의 시작을 알려야함
- start transaction을 실행하면, 그 세션에서는 commit이나 rollback을 만날 때까지 autocommit이 잠시 비활성화됨

> 세션 용어
- 데이터베이스 클라이언트가 데이터베이스 서버에 연결된 순간부터 연결을 종료할 때까지의 논리적인 연결단위를 말함
- 각 세션은 독립적인 작업 공간을 가지며, 한 세션에서 시작된 트랜잭션은 해당 세션 내에서만 유효함

## 트랜잭션의 ACID 속성

### ACID
- 트랜잭션이 보장하는 4가지 속성
- Atomicity: 원자성 - All or Nothing
- Consistency: 일관성
- Isolation: 격리성
- Durability: 지속성

### Atomicity: 원자성
- 트랜잭션은 하나의 원자처럼 더 이상 쪼갤 수 없는 논리적 단위이며, 전부 성공하거나 전부 실패함
    - 모두 성공: commit
    - 모두 실패: rollback

### Consistency: 일관성
- 트랜잭션이 성공적으로 완료되면, 데이터베이스는 항상 일관된 상태를 유지해야함
    - 계좌이체 예시: 트랜잭션 시작 전: 계좌 총액 7만원
        - commit: 총 7만원 보장
        - rollback: 총 7만원 보장
    
    - 제약조건 예시: account테이블에 check(balanced >= 0)이라는 제약 조건이 있다면, 잔고를 음수로 만드는 어떠한 트랜잭션도 결국 실패하고 rollback될 것임
        - 이로써 데이터베이스는 잔고는 0이상이어야함

### Isolation: 격리성
- 하나의 트랜잭션이 실행될 때, 다른 트랜잭션이 해당 트랜잭션의 중간 결과에 끼어들어 간섭할 수 없음
    - 상황: A는 50000원 B는 20000원
        - A가 B에게 1만원을 이체하는 T1을 실행중
        - update로 A의 잔고는 4만원으로 바뀌었지만, commit은 하지 않아 B의 잔고는 2만원
        - 바로 그 순간: 옆 창구에서 은행 직원이 전체 고객의 총잔고를 계산하는 T2를 실행
        - 격리성이 없다면: T2는 A의 잔고를 4만원으로, B의 잔고를 2만원으로 읽어서 총액이 6만원이라는 잘못된 결과를 얻음
        - T1이 아직 완료되지 않은 불안정한 상태를 본 것(더티 리드)
        - 격리성이 있다면: T2는 T1이 아직 commit되지 않았기 때문에 T1이 임시로 변경한 내용을 보지 못함
            - T2는 T1이 시작되기 전의 값, 즉 A의 잔고 5만원과 2만원을 읽어서 정확한 총액 7만원을 계산해냄

### Durability: 지속성
- 성공적으로 완료되어 commit된 트랜잭션의 결과는 시스템에 장애가 발생하더라도 영구적으로 보존됨
    - 상황: A가 B에게 성공적으로 1만원을 이체하고, 은행 앱에서 이체 완료 메시지를 확인
    - 1초 뒤: 은행의 전산 시스템이 다운
    - 지속성 보장: 시스템이 재부팅된 후에도, A의 잔고는 4만원, B의 잔고는 3만원인 상태가 그대로 유지되어 있어야함
        - commit된 결과는 데이터베이스 저장소에 있는 트랜잭션 로그 등에 기록되어, 어떤 장애에도 살아남을 수 있음

### 격리성;의 중요성
- 어느 정도로 엄격하게 격리 시킬 것인가에 따른 격리 수준으로 나눔
- 완벽하게 격리하면 동시에 많은 작업을 처리하기 어려워 성능이 저하
- 낮추면 성능은 상승하지만 데이터 불일치 발생 가능

## 트랜잭션 격리 수준

### 완벽한 격리는 항상 좋은 것일까?
- 완벽한 격리는 데이터 입장에서는 매우 안전함
- 하지만 성능이 매우 좋지 않음
- 트레이드오프
    - 데이터 정합성 vs 동시성/성능
    - 이를 조절할 수 있게 트랜잭션 격리 수준이라는 단계를 제공함

### 동시성 문제
1. 더티 리드(Dirty Read)
    - 한 트랜잭션이 아직 commit하지 않은 데이터를 다른 트랜잭션이 읽는 것
    
2. 반복 불가능 읽기(Non-Repeatable Read)
    - 한 트랜잭션 내에서 똑같은 select 쿼리를 두 번 실행했는데, 그 사이에 다른 트랜잭션이 값을 수정하고 commit하는 바람에 두 쿼리의 결과가 다르게 나오는 현상

3. 유령 읽기(Phantom Read)
    - 한 트랜잭션 내에서 특정 범위의 데이터를 두 번 읽었는데, 첫번째 조회에서는 없었던 새로운 행이 두번째 조회에서 나타나는 현상. 다른 트랜잭션이 새로운 행을 insert하고 commit했기 때문에 발생함
    - 트랜잭션 A가 전자기기 카테고리의 상품 수를 세었더니 5개
    - 잠시후 다른 트랜잭션 B가 새로운 전자기기 상품을 등록해서 6개가 됨
    - 없었던 유령 상품이 나타난 것

### 4가지 표준 격리 수준
- Dirty Read | Non-Repeatable Read | Phantom Read
- READ UNCOMMITTED 
    - 발생
    - 발생
    - 발생

- READ COMMITTED: 커밋된 데이터만 읽을 수 있음(기본 격리 수준)
    - 방지
    - 발생
    - 발생

- REPEATABLE READ: 한 트랜잭션 안에서는 데이터의 일관된 조회를 보장해줌
    - MySQL의 InnoDB 스토리지 엔진이 사용하는 기본 격리 수준
    - 방지
    - 방지
    - 발생(일부 방지)

- SERIALIZABLE: 동시젓성 문제를 완벽하게 차단하지만, 거의 순서대로 실행시켜 동시 처리 성능이 가장 낮음
    - 방지
    - 방지
    - 방지

> 참고: SQL 표준에서는 REPEATABLE READ 수준에서는 팬텀 리드가 발생할 수 있음
- InnoDB 엔진은 대부분의 팬텀 리드가 발생하는 것을 거의 막아줌

### 격리 수준 확인 및 변경하기
- 현재 격리 수준 확인
```sql
select @@transaction_isolation;
-- REPEATABLE READ
```

- 현재 세션에서만 변경하기 (가장 일반적)
```sql
set session transaction isolation level read committed;
```

- 전역으로 변경하기(주의)
```sql
set global transaction isolation level read committed;
```
- 서버 재시작 시에는 사라지는 임시값
- 영구히 하려면 MySQL 설정 파일(my.cnf, my.ini)을 변경해야함

### 실무 팁: 어떤 격리 수준을 선택해야할까?
- 특별한 이유가 없다면 기본 격리 수준을 사용하는 것을 권장
- 언제 그러면 고민할 수 있을까?
    - READ UNCOMIITTED로 낮추는 것을 고려하는 경우
        - 거의 사용하지 않음
        - 언제든 사라질 수 있는 더러운 데이터를 읽는 것을 허용할 수 있기 때문에
        - 사용하는 경우
            - 대용량 데이터에 대한 실시간 집계나 통계 작업을 수행할 때
            - 전자기기에 상품에 대한 대략적인 평균 가격 추이를 실시간으로 보고 싶어하는 경우
            - 몇 상품의 가격이 트랜잭션 도중 잠깐 바뀌었다가 롤백되더라도 전체 평균 가격에 미치는 영향은 아주 미미할 것
            - 정확한 수치가 아닌 경향성을 파악하는 것
        - **가급적 사용하지 말자**
        - 차라리 분석용 데이터베이스를 따로 구축해서 통계 작업을 함

    - READ COMITTED
        - 불필요한 락들을 사용하지 않게 락을 기다리는 시간이 줄어듦
        - 한 트랜잭션에서 똑같은 select를 하는 경우는 거의 없음
        - non-repeatable read가 발생할 가능성이 낮음

    - SERIALIZABLE
        - 데이터의 정합성이 극도로 중요해서 아주 작은 예외도 허용할 수 없을 때 사용
        - 여러 사용자가 동시에 하나의 은행 계좌에서 돈을 인출하거나, 초당 수십건의 주문이 몰리는 한정 수량 상품의 재고를 차감하는 로직이 있다면, SRIALIZABLE을 고려해볼 수 있음(실무에서는 다른 대안을 선택함)
        - 최후의 수단
        - 거의 사용하지 않음
            - select ... for update와 같은 비관적락을 사용 / 버전 정보를 주는 낙관적락
