# S08. 논리적 모델링 4 - 식별 관계, 비식별 관계

## 식별 관계 vs 비식별 관계 - 개념

### 외래 키를 어떻게 사용하는가
- 식별 관계
- 비식별 관계

### 구분하는 기준
- 부모 테이블의 기본키를 물려받아 자식 테이블의 기본 키의 일부로 사용
    - 식별 관계

- 부모 테이블의 기본 키를 단순히 일반 컬럼으로만 사용
    - 비식별 관계

### 식별, 비식별 관계 예시
- 식별 관계 예시
    - 101동 502호
        - 부모의 식별자: 101동
        - 자식의 식별자: (101동, 502호) -> 복합키
    
    - 여기서 자식 502호는 101동 없이는 식별할 수 없음 

- 비식별 관계 예시
    - A 회사, 사원번호 777번
        - 부모의 식별자: 회사 A
        - 자식의 식별자: 777번

    - 부모 없이 식별할 수 자식의 식별자가 독립적으로 식별 가능

### 모델링
- 식별 관계
    - 부모 테이블의 기본 키를 받아서 자식 테이블의 기본 키 + 외래 키로 사용

- 비식별 관계
    - 부모 테이블의 기본 키를 받아서 자식 테이브릐 일반 컬럼 + 외래 키로 사용

## 식별 관계 vs 비식별 관계 - 일대다


### 비식별 관계
- board (1)
    - board_id(PK)

- comment(N)
    - comment_id(PK)
    - board_id(FK)

- 자식 테이블인 comment가 자신만의 고유한 기본 키(comment_id)를 가지는 구조
> ERD에서 비식별 관계는 점선으로 표시. 식별 관계는 실선으로 표시.

- 장점
    - 단순하고 명확함
    - 유연함
    - ORM 친화적

- 단점
    - PK만 봐서 부모와의 관게를 직관적으로 알기 어려움


### 식별 관계
- board(1)
    - board_id(PK)

- comment(N)
    - board_id(PK, FK)
    - comment_no(PK)

- 자식 테이블인 comment가 부모의 기본 키인 board_id를 물려 받아 자신의 기본 키 일부로 삼는 구조
    - comment_no라는 구분을 하나 더 넣어줘야됨(복합 키) - 일대일이 되어버리기 때문

- 장점
    - PK 자체가 비즈니스 의미를 가짐
    - 데이터 정합성 강화
    - 쿼리 최적화 가능성

- 단점
    - 복잡함
    - 유연성이 없음: 댓글을 다른 곳으로 옮길 수 없음
    - 확장성이 떨어짐: 대댓글
    - ORM 매핑 복잡성

> 또한 현재 식별관계는 대리키를 기본 키로 사용하라는 대원칙을 위배 중

---

## 식별 관계의 문제점

### 문제점
- 유연성 부족
- 기본 키 컬럼의 전파와 복잡성 증가
    - 다른 엔티티가 해당 엔티티를 또 참조하는 경우 또 뭐가 추가되고
    - 조인도 힘듦
> 현대는 비식별 관계를 주로 사용함

### 식별 관계의 장점
- 조인 없이 데이터를 조회하기 용이함

### 결론
- 일대다에서는 기본적으로 비식별 관계를 사용하자!!
- **대리 키 + 비식별 관계**
- 대리키: PK
- 자연키: UNIQUE
- 관계: 비식별

## 식별 관계, 비식별 관계 - SQL 쿼리, 성능

### 단순 SQL 쿼리
- 요구사항: 특정 댓글 하나의 내용을 조회하라

- 비식별 관계 쿼리
```sql
select content from comment_non_identifying where comment_id = 500;
```

- 식별 관계 쿼리
```sql
select content from comment_identifying where board_id = 10 and comment_no = 3;
```

### SQL 조인과 조회
- board_id가 1인 대댓글을 찾아라

- 비식별 관계
    - 조인이 필요함
    - 인덱스: 조인을 위한 인덱스가 작고 단순함

- 식별 관계: 조인 없이 데이터를 찾을 수 있음
    - reply 테이블만 조회하면 되므로 매우 빠름
    - 인덱스: 크고 무거운 복합 인덱스
    
### 단일 식별자의 장점: 단순함이 주는 강력함
- API 설게 URL 설계의 단순성
- 개발 편의성 및 실수 방지
- 객체지향 및 ORM과의 조화

### 식별 관계 vs 비식별 관계: 인덱스 활용
- 요구사항
    - 10번 게시글에 달린 모든 댓글을 조회하라

- 비식별 관계의 인덱스 전략
    - board_id에 인덱스를 걸어줘야함

- 식별 관계의 인덱스 전략
    - (board_id, comment_no)로 인덱스가 걸려 있음
    - 복합 인덱스는 첫번째 컬럼의 인덱스로 검색할 때 효율적으로 동작함
    
- 결론
    - 식별 관계는 부모 - 자식 순으로 데이터를 조회하는 특정 시나리오에서 별도의 인덱스 생성 없이도 높은 성능을 보일 수 있음
    - 비식별 관계는 FK 컬럼에 인덱스를 직접 생성해야만 동일한 성능을 확보할 수 있음

## 식별 관계 vs 비식별 관계 - 일대일

### 비식별 관계
- member
    - member_id(PK)

- member_detail
    - member_detail_id(PK)
    - member_id(FK, UNIQUE)

### 식별 관계
- member
    - member_id(PK)

- member_detail
    - member_id(PK, FK, UNIQUE)

### 장단점
- 비식별 관계
    - 장점
        - 독립성과 유연성
        - 일관성 있는 설계

    - 단점
        - 일대일을 보장하기 위해 UNIQUE가 추가로 필요함

- 식별 관계
    - 장점
        - 강한 논리적 결합 표현
        - 단순함

    - 단점
        - 낮은 유연성

### 결론 및 실무 가이드
- 특별한 경우가 아니라면 비식별 관계를 사용하는 것을 권장
- 유연성과 확장성이 매우 중요함
- 식별 관계는 분리는 했지만 완벽하게 하나의 테이블로 취급해야할 때 사용

### 왜 비식별을 기본으로 가는 것일까?
- 일대일 => 일대다로 변경해야한다면
    - 비식별은 UNIQUE 제약조건만 해제하면 됨
    - 식별은 member_id를 PK로 사용하고 있어서 UNIQUE를 기본으로 깔고 들어감
        - PK를 결국 바꿔야함

## 식별 관계 vs 비식별 관계 - 다대다 관계 (1:N)
- 기본 비즈니스 규칙
    - 청바지 2개
    - 청바지 3개
    - 이런식으로 되면 안됨

### 식별 관계(order_item)
- order_item
    - order_id (FK, PK)
    - product_id (FK, PK)
    - 복합키

- 장점
    - **비즈니스 규칙을 강제함**
    - 인덱스 효율

- 단점
    - 유연성 부족:
        - 하나의 주문에 옵션이 다른 동일 상품을 여러 개 담을 수 있다라는 규칙이 추가되면 지금 구조로는 안됨
    - 복잡성 증가

### 비식별 관계
- order_item
    - order_item_id(PK)
    - order_id(FK)
    - product_id(FK)

- 문제: 지금 상태로는 비즈니스 규칙에 위배됨
- 해결: UNIQUE 제약을 걸음 UNIQUE(order_id, product_id)

- 장점
    - 유연성과 데이터 정합성을 모두 확보

---

## 식별 관계 vs 비식별 관계 - 다대다 2

### 확장성 문제
- 새로운 요구사항: 옵션이 다르면 별개의 상품으로 주문할 수 있게 해라
    - (order_id, product_id) => (order_id, product_id, option_id)로 변경되어야함

### 비식별 관계의 경우
- 새로운 컬럼 넣고,
- UNIQUE를 다시 걸면 됨
> 참고: MySQL은 NULL을 서로 다른 값으로 취급하므로 동일 주문, 상품, NULL이 된다는 점을 주의하자!!!

### 식별 관계의 경우
- 결국 PK를 건들여야함

## 식별 관계 vs 비식별 관계 - 현대적인 설계 트랜드

### 현대 트렌드
- **비식별 관계**
    - 압도적인 유연성과 민첩성
    - 단순성과 일관성
    - ORM 친화성
    - 비즈니스 로직과의 완벽한 분리

### 식별 관계를 고려해야하는 경우
- 자식 테이블이 부모 테이블의 존재에 완벽하게 종속적일 때
- 자식 테이블의 데이터가 독립적으로 조회될 일이 절대 없고, 항상 부모를 통해서만 접근될 때
- 향후 비즈니스 요구사항이 바뀔 가능성이 절대 없을 때

### 최종 정리
- 대리키: PK
- 자연키: UNIQUE
- 관계: 비식별
