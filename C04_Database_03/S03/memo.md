# S03. 개념적 모델링

## 요구 사항 분석과 핵심 요소 식별

### 요구 사항 분석
- 회원 기능
    - 회원은 고유한 아이디, 비밀번호, 회원명, 주소(기본 배송지), 연락처 정보를 이용해 가입할 수 있음
    - 회원은 자신의 정보를 언제든지 수정할 수 있음
    - 회원은 시스템에서 탈퇴할 수 있음

- 상품 기능
    - 고나리자(또는 판매자)는 상품을 시스템에 등록할 수 있음
    - 상품은 고유한 상품 코드, 상품명, 상품 가격, 재고 수량을 정보로 가짐
    - 상품의 가격이나 재고 등은 언제든지 변경될 수 있음

- 주문 기능
    - 회원은 여러 상품을 한 번에 주문할 수 있음
    - 주문 시, 각 상품의 주문 수량을 지정할 수 있음
    - 주문 시, 배송지를 변경할 수 있음 (기본 배송지와 다를 수 있음)
    - 주문이 완료되면, 주문은 '주문 완료' 상태가 되며, 고유한 주문 번호가 생성됨
    - 주문에는 주문 일시(주문한 날짜), 최종 결제 금액, 배송지 정보가 기록되어야 함
    - 관리자는 주문 상태를 '배송 중', '배송 완료' 등으로 변경할 수 있음
    - 회원은 자신의 주문 내역을 목록으로 조회할 수 있음

### 뼈대를 찾아내는 방법 - 명사, 동사
- 명사: 우리가 관리해야할 데이터의 대상(엔티티, 속성)
- 동사: 데이터들 사이의 행위나 관계(관계)

### 핵심 데이터 덩어리, 엔티티 도출하기
- 가장 기본적인 요소, 엔티티
- 회원, 상품, 주문
    - 주문하다는 동사인데?
    - 주문하다라는 행위의 결과물인 주문서를 의미함

### 결과
- 회원 <-> 주문 <-> 상품

## 엔티티란

### 엔티티
- 데이터를 저장하고 관리해야할 대상
- 저장할만한 가치가 있는 정보를 여러 개 가지고 있으면서, 다른 것과 명확히 구분되는 유무형의 모든 것을 말함

### 유형 엔티티
- 직접 만지거나 볼 수 있는 대상
    - 회원, 직원
    - 상품, 책
    - 건물

### 무형 엔티티
- 물리적인 형태는 없지만, 개념적으로 존재하는 관리 대상을 의미
- 주로 사건이나 개념
    - 주문, 예약, 수강 신청: 행위(동사)의 결과로 생성되는 사건을 기록하는 엔티티
        - 주문 번호, 예약 날짜와 같은 속성을 통해 관리됨
    - 계좌, 학과: 물리적 실체는 없지만, 은행 업무나 학사 관리에서 중요한 역할을 하는 개념적인 엔티티

## 엔티티 분류 1

### 필요성
- 회원과 주문 테이블의 성격이 다름
- 회원은 한 번 생성되면 변경이 없고, 그 데이터의 양도 주문에 비해 적음
- 주문은 시간이 지남에 따라 폭발적으로 데이터가 증가할 수 있음을 예측할 수 있음

### 존재 형태에 따른 분류
- 유형 엔티티
    - 사람

- 개념 엔티티
    - 부서

- 사건 엔티티 (이벤트 엔티티)
    - 업무 프로세스가 진행됨에 따라 발생하는 특정 행위나 사건을 표현
    - 주문, 계약, 취소, 사고 접수

### 알 수 있는 사실
- 사건 엔티티는 그 수가 폭발적으로 증가 될 수 있음
- 전략을 세울 수 있음
    - 인덱스 전략(조회)
    - 데이터 파티셔닝 및 아키이빙
        - 현재까지의 모든 기록을 남겨놓으면 비효율적
        - 파티셔닝
            - 주문 테이블을 주문 일시를 기준으로 물리적으로 분리하여 저장
            - 최근 한 달 주문 조회와 같은 쿼리는 훨씬 작은 데이터 영역만 탐색하므로 속도가 향상
        - 아카이빙
            - 사용 빈도가 극히 낮은 오래된 데이터를 별도의 백업 테이블이나 스토리지로 옮겨서, 현재 운영 중인 테이블의 크기를 작고 빠르게 유지하는 전략

### 역할 및 발생 시점에 따른 분류
- 기본 엔티티
    - 다른 엔티티에 의해 생성되지 않고 독립적으로 존재할 수 있는 핵심적인 엔티티(키 엔티티)
    - 자신만의 고유한 주 식별자를 가짐
    - 독립적으로 생성이 가능
    - 주로 다른 엔티티의 부모 역할을 함
    - 회원, 상품, 사원, 부서, 고객, 자재

- 중심 엔티티
    - 기본 엔티티로부터 파생되어 생성됨
    - 행위 엔티티를 생성하는 허브 역할을 함
    - 기본 엔티티로부터 발생됨
    - 업무 프로세스의 중심적인 역할을 담당함
    - 데이터 발생량이 많고, 많은 행위 엔티티를 파생
    - 주문, 계약, 청구, 매출(누가 했는지, 얼마인지, 몇 개인지 등등)
        - 주문은 진짜 주문에 대한 것만(구매자, 주문 번호, 주문 일자, 금액)

- 행위 엔티티
    - 주로 기본 엔티티와 중심 엔티티로부터 ㅂ라생
    - 업무가 흘러가면서 생성되는 상세 정보를 기록하는 엔티티
    - 내용이 자주 변경되거나 데이터양이 지속적으로 증가하는 특징을 가짐
    - 모델 내에서 데이터 양이 가장 많은 엔티티가 되는 경우가 많음
    - 주문 이력, 신청변경이력, 결제 내역, 로그, 주문 항목
        - 주문 이력은 좀 더 상세 정보(주문상세번호, 주문 번호, 상품 코드, 수량, 구매 가격)
        - 즉, 주문 번호에 각각의 상품들이 뭐가 되었는지 등(중복해서 가능)

- 회원 - 주문(회원, 주문번호) - 주문 이력(주문번호, 주문상세번호, 상품 코드)

## 엔티티 분류 2

### 강한 엔티티과 약한 엔티티
- 강한 엔티티
- 다른 엔티티의 존재 여부와 관계없이 독립적으로 존재할 수 있는 엔티티를 말함
- 고유한 주 식별자(기본키)를 가짐

- 약한 엔티티
- 다른 엔티티가 존재하지 않으면 존재할 수 없은 종속적인 엔티티
- 자신을 식별하기 위해 소유 엔티티의 주식별자를 빌려옴
- (부모의 기본키 / 자식의 속성) 복합 기본키를 가짐

### 구조적 관계 표현을 위한 특수 엔티티
- 연관 엔티티
    - 두 개 이상의 엔티티 간에 발생하는 특정 사건, 행위, 계약 등을 포함하기 위해 도출되는 엔티티
    - 즉, 관게 자체가 중요한 속성을 가져 하나의 독립된 실체로 다뤄줘야할 때 사용됨
    - 개념적 본질: 
        - 관게 종속되는 속성을 저장하기 위함
        - 학생과 과목의 관계에서는 성적이나 수강신청일 같은 데이터가 발생
        - 학생, 과목의 것도 아닌 수강이라는 행위 자체에 속함
    - 구조적 의미:
        - 두 엔티티 간의 다대다 관계를 해소함
        - 관계형 데이터베이스는 다대다 관계를 직접 구현할 수 없어 중간 엔티티를 통해 일대다 다대일로 변환함

- 슈퍼타입/서브타입 엔티티
    - IS-A 관계(관리자는 사원의 한 종류)
    - 슈퍼타입: (부모)
        - 그룹 내 모든 엔티티들이 공통적으로 가지는 속성과 관계를 정의하는 상위 엔티티
    - 서브타입: (자식)
        - 슈퍼타입으로 공통 속성을 상속 받고, 자신만의 고유한 속성이나 관계를 추가로 가지는 하위 엔티티
    - 사원 - (정규직 사원 / 계약직 사원)

## 카디널리티와 참여도

### 관계의 수량, 카디널리티 분석하기
- 1:1, 1:N, N:1: N:M

- 표기 기준 정리
    - 엔티티 인스턴스(행) 하나가 다른 엔티티의 인스턴스와 맺을 수 있는 최대 숫자를 기준
        - 1: 한 엔티티의 인스턴스가 다른 엔티티의 인스턴스와 최대 1개의 관계를 가질 수 있음
        - N: 한 엔티티의 인스턴스가 다른 엔티티의 인스턴스와 초대 N개의 관계를 가질 수 있음

### 카디널리티 정의하는 방법
- 먼저 ? -> (1, N)을 파악
- 이후 뒤집어서 (1, N) -> (1, M)을 파악
- 즉 엔티티 하나씩 봐야함

### 관계의 필수 여부, 참여도 분석하기
- 한 엔티티의 인스턴스가 관계에 반드시 참여해야하는지(필수), 아니면 참여하지 않을 수도 있는지를 결정
- 회원 입장에서 주문과의 관계는 필수? 선택? => 선택
    - 참여는 선택(Optional)
- 주문 입장에서 회원과이 관계는 필수? 선택? => 필수
    - 참여는 필수(Mandatory)

### 관계의 표현과 외래 키
- 개념적 모델링 단계에서는 엔티티 간의 관계를 관계선으로 표현하며, 외래 키 같은 특정 기술의 구현적 요소는 포함하지 않는 것이 원칙
    - 주문에 회원 id를 포함하지 않음

## ERD 완성하기

### 표기법의 이해와 실무 선택
- 피터 챈 표기법
    - 엔티티는 사각현
    - 관계는 마름모
    - 속성은 타원

- 까마귀발 표기법(사실상 표준)
    - 닭발: N
    - |: 1
    - o: 선택적 관계 / | 필수적 관계

- 조합
    - o|: 0개 또는 1개
    - ||: 반드시 한개
    - o<: 0개 또는 여러 개
    - |<: 1개 또는 여러 개

## 연관 엔티티 - 다대다 관계 해결

### 문제 1: M:N 관계는 물리적으로 구현할 수 없음 - 주문과 상품(N:M)
- 시도 1: 1:N 방식
    - 주문 테이블
        - (주문 id(PK), 회원 id, 주문 일시, 상품 id)
        - (1001, 1, 2025-08-06, 101)
        - (1001, 1, 2025-08-09, 102)
        - PK 충돌이여서 만들 수 없음

- 시도 2: 컬럼 계속 추가하기
    - (주문 id, 회원 id, 주문 일시, 상품 id 1, 상품 id 2, ...)
    - 확장성 부재
    - 공간 낭비
    - 데이터 검색의 어려움

- 시도 3: 한 컬럼에 여러 값을 욱여넣기 (절대 피해야 할 방식)
    - (주문id, 회원 id, 주문 일시, **상품 ids**)
    - 데이터 검색의 어려움
    - 데이터 수정 및 삭제의 복잡성
    - 원자성 위반
        - 데이터베이스의 제 1정규형, 테이블의 모든 칸은 원자적인 값 하나만 가져야함을 위반함

### 해결책: 관계를 엔티티로 승격하자
- 다대다 => 일대다 다대일로 풀어내기
- 주문 항목
    - 주문 항목 번호
    - 주문 가격
    - 주문 수량

- 주문 - 주문 항목 (1:N) - 양방향 필수적 관계
- 주문 항목 - 상품 (N:1) - 주문 항목은 선택적

### 결과
- 주문 테이블 (중심 엔티티)
    - 주문 id(PK), 회원 id(FK), 주문 일시

- 주문 항목 테이블 (행위 엔티티 - 폭발적인 데이터량 증가 예측 가능)
    - 주문 항목 id(PK), 주문 id(FK), 상품 id(FK), 주문 가격, 주문 수량

- 상품 테이블 (기본 엔티티)
    - 상품 id(PK), 상품명, 상품 가격

### 실무 팁: - M:N은 반드시 풀고, 필요한 속성을 찾아라
- 추가적인 속성을 가질 확률이 매우 높음(JPA 다대다 쓰지 말라는 이유와 동일)

## 용어 사전

### 필요성
- 프로젝트의 모든 이해 관계자들은 모두 같은 비즈니스 용어를 사용하는 것이 중요함
- 개념적 모델링 단계에서는 이 한글 용어들을 정한 규칙에 따라 영어로 정확하게 변환해야함

### 가이드
- 단일어 중심
    - member
    - product
    - id
- 복합어를 사용하지 말 것
