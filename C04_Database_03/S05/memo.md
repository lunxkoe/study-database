# S05. 논리적 모델링 - 키

## 논리적 모델링
- 관계형 데이터베이스 구조에 맞게 변환
- 특정 관계형 데이터베이스에 종속되지 않게 하는 것이 초점
- 이 때, 가장 중요하게 다루어야 할 주제가 **키**

## 다양한 종류의 키
- 기본 키
- 테이블의 모든 행을 유일하게 식별하는 대표키
    - NULL 값을 가질 수 없음
    - 반드시 유일해야함
    - 값이 변하지 않아야함

- 후보 키
- 기본키가 될 수 있는 자격
    - 유일성: 모든 행을 서로 구분할 수 있어야함
    - 최소성: 행을 유일하게 식별하는데 꼭 필요한 최소한의 컬럼만 포함

- 대체 키
- 후보 키 중에서 기본 키로 선택되지 않은 나머지 키들을 말함
- member_id를 기본 키로 선택 시, 후보 키였던 이메일, 연락처는 대체 키

- 외래 키
- 가장 중요한 키로, 테이블 간의 관계를 연결하는 역할
- 한 테이블의 컬럼이 다른 테이블의 기본 키를 참조하는 역할

---

## 자연 키 vs 대체 키 1 - 자연 키

### 자연 키란?
- 자연스럽게 발생하는, 의미를 가진 데이터를 기본 키로 사용하는 것을 말함
- 주민등록번호, ISBM

- 장점
    - 직관성
    - 중복 방지

### 치명적인 약점: 변경이라는 시한 폭탄
- 변경 가능성
- 기본 키는 영원히 변하지 않아야 함
- 하지만 비즈니스 로직에 종속되는 자연 키는 이 원칙을 지키키기가 매우 어려움

- 시나리오: 회원의 이메일을 PK로 사용한 쇼핑몰의 재앙

### 재앙
- 이메일을 기본 키로 했다면, 이메일 변경이 불가능하게 되는 문제가 발생

---

## 대리 키

### 대리 키란
- 비즈니스 로직과 완전히 무관한, 오직 데이터를 식별하기 위한 용도로만 존재하는, 임의의 값을 기본 키로 사용하는 것
- **절대 변하지 않는 것**
- **보통 순차적으로 증가하는 정수나 UUID를 사용함**

### 자연 키의 올바른 사용법: UNIQUE 제약 조건
- 행을 유일하게 식별할 수 있음
- 데이터 무결성: 중복을 데이터베이스 차원에서 막아줄 수 있음
- 조회 성능: unique는 인덱스가 자동으로 생성되면서 email을 통한 조회 성능도 보장됨

### 표준 패턴
- 대리 키: 기본 키로 사용
- 자연 키: UNIQUE 제약조건 활용

---

## 자연 키 vs 대리 키 3 - 성능 트레이드 오프

### 자연 키 사용의 장점
- 조인 없이 해당 테이블에서 원하는 정보를 찾을 수 있음
- 주문 정보에서 사용자의 이메일을 알고 싶은 경우

### 자연 키 사용의 단점
- 외래 키 크기
    - 공간 낭비
    - 성능 저하

- 인덱스 단편화와 쓰기 성능 저하

### 대리 키 사용의 장점
- 뛰어난 쓰기 성능: Auto_increment나 sequence를 사용하는 경우 이전 값보다 항상 큰 숫자를 넣음
- 외래키 크기 최소화

### 대리 키 사용의 단점
- 추가 조인 발생
- 추가 인덱스 생성
    - PK 인덱스
    - UNIQUE 인덱스

---

## 자연 키 vs 대리 키 4 - 현대적인 설계

### 핵심
- 과거: 자연 키 
- 현재: 대리 키: 느슨한 결합

### 결론
- 대리키 - PK
- 자연키 - UNIQUE

---

## 복합키 설계

### 개념
- 하나의 컬럼만으로는 행을 고유하게 식별할 수 없는 상황

### 필요성
- 전제: 현재는 대리키가 아닌 자연키를 사용해서 구성하고 싶은 경우를 말함
- 필요성 대두: 영화
    - 상영 정보: 제목으로는 안됨(영화 제목, 사영 시작 시간)
    - 좌석 예배: 어떤 상영의 어떤 좌석인지를 명시

### 만드는 법
```sql
primary key (movie_title, screening_at, seat_number)
```

### 자연 키를 복합키로 사용하는 예시에서 발견되는 문제점
- 변경 가능으로 인한 불안정성
- 외래 키 참조의 복잡성과 크기 증가 
    - 조인 복잡

### 대안
- 대리 키!!
- 대신!!
    - 자연 키 부분을 조합해서 (movie_title, screening_dt, seat_number)를 UNIQUE KEY로 만들어야함

### 결론
- 대리 키 사용
- 자연 키 조합에 UNIQUE를 걸기

## 다대다 관계와 복합 키

### 문제 상황: 다대다 관계 모델링
- 다른 테이블의 대리 키(외래 키)들을 조합해서 복합키를 만드는 경우
- 주문과 상품

### 다대다 관계 해소를 위한 연결 테이블
- orders(1) - (N)order_item(N) - (1)product
- 질문: order_item 테이블의 기본 키를 뭐로 해야할까?
    - {order_id, product_id}: 복합 키
    - {order_item_id}: 대리 키 / {order_id, product_id}: UNIQUE

### 복합 키 장단점
- 장점
    - 데이터 무결성 보장
        - 하나의 주문에 동일한 상품이 두 번 이상 들어가는 것을 원천적으로 차단함
    - 논리적 명확성

- 단점
    - PK가 너무 뚱뚱함
    - 확장성의 제약
    - ORM 사용 제약

### 대리 키 + UNIQUE 조합 사용
- 장점
    - 유연하고 일관된 참조
    - 데이터 무결성 유지
    - 개발 편의성

- 단점
    - 약간의 오버헤드: 인덱스 생성(기본 키(대리 키) + UNIQUE)

### 결론
- 대리 키 
- UNIQUE 활용
