# 논리적 모델링 3 - 일대일, 다대다 관계

## 일대일 관계 - 시작

### 필요성
- 보통 일대일 관계는 하나로 합치는 것도 가능해서 그렇게 하는 경우가 많음
- 단순한 시스템 보다는 좀 더 복잡한 시스템에서 주로 사용
    - 다 때려박으면 성능 문제가 생길 수도 있음
    - 이때 분리하여 성능을 최적화할 수 있음
    - member / member_details
    - 선택적 정보 저장
        - 판매자만 따로 뭔가를 더 추가 해야할 경우
        - 모든 사용자(판매하지 않는 사람 포함)에 컬럼을 추가하면 비용 발생

- 비즈니스 모델에 따른 분리

### 장점
- 관심사의 분리
- 독립적인 확장성

### 일대일 관계 만들기
- 회원, 회원 상세 정보
- member, member_details
- member_details에 member_id를 외래 키
    - 문제: 사실 이건 일대일이 아님 - 어떻게 해야 강제할 수 있을까?
    - UNIQUE 조건을 걸어줘야함!!!!!

> 일대일 관계와 식별 관계
- 부모 테이블의 PK를 자식 테이블의 PK+FK로 사용하는 방법도 있음(식별 관계라고 함)

## 일대일 관계 - 외래 키 위치

### 어디에 두어야 할까?
- 방법1: 보조 테이블(member_detail) / 보조 테이블 FK -> 주 테이블 PK 참조
- 방법2: 주 테이블(member) / 주 테이블 FK -> 보조 테이블의 PK 참조
> 결론: 특별한 이유가 없다면 보조 테이블에 외래 키(UNIQUE)를 걸어주는 것이 좋음

### 방법 1
- 장점
    - 선택적 관계 표현의 용이성
    - 관계 확장의 유연성

- 단점
    - 관계 존재 여부 확인: 회원 상세 정보가 있는지를 member로써는 알 수 없음
    - member_detail과 조인을 해야알 수 있음

### 방법 2
- 방법
    - 상세 정보가 없을 수도 있으면 NULL / 없으면 NOT NULL + UNIQUE

- 장점
    - 주 테이블 중심 설계: 주 테이블만 조회해도 보조 정보의 존재 여부를 바로 확인할 수 있음
    - 관계 존재 여부 확인의 용이성

- 단점
    - 나쁜 확장성
    - 관계 변경 유연성: 일대다로 전환 어려움
    - NULL 값 허용
    - 데이터 추가의 복잡성

### 방법 1 사용

## 일대일 관계 - 주 테이블에 FK

### 고려해볼 수 있는 상황
- 관계가 선택이 아닌 필수일 때
    - member_detail_id에 NOT NULL을 걸어 데이터 무결성을 보장할 수 있음
    - 존재하지 않는 member_detail_id가 들어오는 것도 막아줌

- 조회 성능이 극도로 중요할 때
    - 주 테이블만 조회해도 보조 정보의 존재 여부나 식별자를 알아야하는 경우
    - 보조 테이블에 있는 경우는 조인을 해야지만 알 수 있음
    - 주 테이블에 있는 경우는 member만 조회하면 알 수 있음

> 섣부른 최적화는 금물
- 방법 1의 조인은 문제가 되지 않음
- 성능 최적화의 최후의 보루로 선택해야함(인덱스 설계 같은 것이 더 좋음)

### 결론
- 핵심 원칙
    - 기본 선택: 보조 테이블에 외래 키를 둔다

    - 예외적 선택: 주 테이블에 외래 키를 둔다
        - 두 테이블의 관계가 필수적일 때
        - 조인을 피해야할 만큰 조회 성능이 매우 중요할 때
        - 특정 ORM 매핑의 편의성이 우선될 때

## 다대다 관계 - 관계형 모델의 한계

### 두 테이블로 다대다를 표현할 수 없는 이유
- PK는 테이블에 단 하나의 행에만 존재해야만 함!!

### Java
- 객체에 컬렉션을 가지도록 해서 표현할 수 있음

### NoSQL
- 문서 안에 다른 데이터를 배열 형태로 포함할 수 있음

---

## 다대다 관계 - 연결 테이블

### 다대다 관계의 해결법 - 연결 테이블
- orders <-> order_product <-> product

- orders
    - orders_id(PK)

- order_product
    - order_product_id(PK)
    - orders_id(FK)
    - product_id(FK)
    - (orders_id, product_id) UNIQUE

- product
    - product_id(PK)

---

## 다대다 관계 - 속성

### 관계의 속성과 연결 테이블
- 연결 테이블은 관계 자체에 대한 추가적인 속성을 가지게 됨
    - JPA에서 다대다 사용하지 말라고 한 이유
- 개념적 모델링에서 이 연결 테이블을 설계해야함

### 네이밍
- order_product
    - 이건느 딱 연결 테이블 그 자체만의 역할을 위한 네이밍

- 만약 추가적인 컬럼 order_price, order_quantity가 추가된다면
    - order_item으로 해야함

### order_item
- 비즈니스 적인 연결 테이블의 의미를 가지게 됨
- order_product: 기술적으로 다대다 관계를 풀어내는 것에 초점
- order_item: 비즈니스 속성과 의미를 가짐. 기술적으로 다대다 관계도 풀어냄

### 식별, 비식별 관계
- order_item은 order_item_id를 기본 키로 사용하는 비식별 관계
- (order_id, product_id)를 복합 기본 키로 사용하는 식별관계 모델이 있음
- 두 가지 방식의 차이점과 각각의 장단점은 매우 중요함. 이후 다룸

## 다대다 관계 - 개념적, 논리적 모델링

### 속성이 없는 다대다 관계 예시
- 개념적 모델링
    그냥 딱 다대다 연결 표시만 하고 끝내면 됨

- 논리적 모델링
    - 연결 테이블을 반드시 만들어야 함

### 속성이 있는 다대다 관계 예시
- 개념적 모델링
    - 행위 자체에 저장해야할 중요한 정보가 있다는 뜻
    - 관계 자체가 하나의 엔티티처럼 취급되어야함
    - 이를 연관 엔티티라고 하고 개념적 모델링 단계에서 튀어나와야함

- 논리적 모델링
    - 동일하게 연결 테이블(연관 테이블)이 튀어나와야함

### 이론과 실무의 차이: 속성 없는 다대다 관계
- 실무에서는 속성 없는 다대다 관계는 거의 나오지 않음
- 항상 그 자체로 중요한 비즈니스 개념으로 나오지 않는지 항상 고민해야함(다대다 관계 발견 시)
