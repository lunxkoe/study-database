# 물리적 모델링

## 물리적 모델링 개요

### 물리적 모델링이란?
- 논리적 모델을 특정 데이터베이스 관리 시스템의 특성과 성능을 고려하여 구제적인 저장 구조, 물리적 스키마로 변환하는 과정
- **성능 / 특정 DBMS**

### 물리적 모델링 프로세스
- 테이블과 컬럼 변환
- 데이터 타입 정의
- 제약 조건 설정
- 인덱스 설계
- 역정규화 및 성능 튜닝
- 파티셔닝, 샤딩 등 기타 기법 적용
- 뷰, 저장 프로시저, 함수, 트리거 생성

## 테이블과 컬럼 변환 규칙 1 - 기본 규칙

### 일반적인 명명 규칙
- 영어 사용
- 소문자 스네이크 케이스
- 명확하고 서술적인 이름
- 일관성 있는 접두사와 접미사(member_id, product_name)
- 예약어 사용 금지: order(X) -> orders

### 컬럼 이름 규칙
- 기본 키는 테이블명_id
- 외래 키는 참조하는 테이블의 PK 명으로 그대로 사용
- 날짜/시간 컬럼은 접미사로 용도를 명확히 함
    - _at: 특정 작업이 발생한 정확한 시점
    - _datetime, _dt(not date): 날짜와 시간
    - _date: 날짜
    - _time: 시간

- 불리언은 is_, has_
- 컬럼은 단수 명사 사용(테이블 명은 복수를 사용하는 경우도 있음)
- 의미 있는 축약어
    - 약속된 축약어의 경우는 사용
- 데이터의 성격을 명확히 표현
    - 단위나 종류 명시
        - _kg
        - _g
        - _cm
        - _m
    - 문맥 제공
        - product_name
        - product 테이블의 name도 나쁘진 않음
        - 다만 여러 테이블에서 자주 사용되는 용어라면 구분을 고려

> 가장 중요한 것은 **일관성**

## 데이터 타입 1 - 문자, 숫자, PK 타입

### 문자열 타입
- VARCHAR(M): 가변 길이 문자열, M은 저장할 수 있는 최대 길이를 의미함
    - 실제로 5글자를 사용해도 해당하는 공간 + 길이 정보(1-2바이트)정도 사용함
    - 장점: 공간 효율이 좋음
    - 단점: 길이가 변하기 때문에 데이터 수정 시 추가적인 작업이 필요할 수 있음
    - 용도: 이름, 제목, 주소 등 대부분의 문자열 데이터에 사용됨

- CHAR(M): 고정 길이 문자열
    - 5글자를 써도 M=10이면 10의 공간을 차지함
    - 장점: 길이가 고정되어있어 데이터 처리 속도가 빠름
    - 공간 낭비가 쉬움
    - 용도: 주민등록번호, 전화번호, 성별처럼 길이가 항상 고정된 데이터에 사용할 수 있음

- TEXT: 매우 긴 텍스트를 저장할 때 사용
    - TEXT < MEDUUMTEXT < LONGTEXT
    - 게시물의 본문, 상품의 상세 설명 등에 사용됨

### 실무 가이드
- 일단 VARCHAR를 사용
- 단, 합리적으로 전체 길이를 고민해야함

### 심화: VARCHAR 길이는 왜 메모리 사용량에 영향을 줄까?
- 디스크 저장 공간과 메모리 사용은 다른 이야기이기 때문
- MySQL이 쿼리를 처리하기 위해 메모리에 임시 공간을 할당할 때 VARCHAR에 설정된 최대 길이를 기준으로 삼는 경우가 많음
- **합리적인 최대 길이를 설정하자**

### 숫자 타입
- 정수형과 소수형

### 실무 가이드
- TINYINT는 상태 값 처리에 주로 사용
- UNSIGNED 옵션을 활용해서 양수만 저장도 가능 
- DECIMAL(M, D): 고정 소수점, 아주 정확한 연산이 필요하다면 무조건 사용
- DOUBLE, FLOAST: 부동 소수점, 미세한 오차가 생길 수 있음, 돈과 관련된 계산에서는 절대 사용하면 안됨

### PK 타입 선정: INT vs BIGINT
- 실무 가이드: BIGINT 사용을 권장함
- 이유: 현대에는 차이가 생각보다 작음(기술의 발전)

## 데이터 타입 2 - 날짜와 시간 타입

### 실무 가이드
- 날짜 + 시간: datetime(timestamp X)
- 날짜: date
- 시간: time 

### 생서일과 수정일 자동화
- default current_timestamp: 컬럼에 기본값을 명시하지 않으면 null인 경우 자동으로 값을 넣어줌
- on update current_timestamp: 해당 행이 수정될 때마다 데이터베이스가 자동으로 값을 넣어줌

### 비즈니스 날짜 vs 자동 생성 날짜
- ordered_at: 고객이 주문을 한 시간
- creeate_at: 실제 디비에 저장이 된 시간

### 기타 타입
- JSON: JSON 형식을 그대로 저장할 수 있음
    - 남용은 금지

## 역정규화

### 필요성: 성능
- 정규화의 가장 큰 단점은 너무 많은 테이블이 생성되어 많은 조인이 발생할 수 있음
- **데이터의 조회 성능을 향상시키기 위해, 의도적으로 데이터 모델의 정규화 원칙을 위반하여 데이터의 중복을 허용하는 프로세스**

### 실무 역정규화 기법
- 중복 컬럼 추가
    - 조회시 필요한 다른 테이블의 컬럼을 그대로 복사해서 가져오는 것
    - 주의할 점: 
        - 데이터 불일치 가능성이 있음
        - 주문 당시의 상품명은 역사이기 때문에 불변이 타당할 때도 있음

- 파생 컬럼 추가 (계산된 값의 저장)
    - 총 주문 금액
    - member table에 총 주문 금액을 같이 저장함
    - 주의할 점:
        - 쓰기 작업의 부하를 증가시킴

- 테이블 통합 및 분할
    

### 역정규화 시 데이터 일관성 유지 방안
- 애플리케이션 로직
- 데이터베이스 트리거
- 배치 작업: 실시간이 그렇게 중요하지 않은 경우 사용

### 역정규화 언제 할까?
- 섣불리 적용하지 마라
- 데이터로 증명하라
    - 단순히 느리다 X -> 실제 성능 측정을 통해 병목이 되는 쿼리를 명확히 식별
- 읽기/쓰기 비율을 고려하라
- 비용을 계산하라

### 용어 정리
- 역정규화: 이미 정규화된 모델을 성능 등의 이유로 의도적으로 되돌리는 과정을 표현하는 용어
- 반정규화: 정규화에 반대하는 의미를 가짐
- 비정규화: 정규화가 전혀 적용되지 않은 상태나 구조를 말함
