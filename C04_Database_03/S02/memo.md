# S02. 설계의 중요성

## 잘못된 설계가 부르는 재앙

### 나쁜 설계의 3대 문제점
- 데이터 무결성 훼손 | 성능 저하 | 유지보수 비용 증가

### 데이터 무결성 훼손 (신뢰도 하락)
- 중복 저장되는 데이터가 많이 있음
- 수정 이상 (Update Anomaly)
- 삽입 이상 (Insertion Anomaly)
- 삭제 이상 (Deletion Anomaly)
    - 주문 정보가 1개였다면 삭제되면 고객 정보 자체가 사라져버림

### 성능 저하 (속도 저하)
- 느려지는 조회 속도
    - 풀 테이블 스캔

### 유지보수 비용 증가 (확장성 저하)
- 작은 변경의 큰 파급효과

## 설계의 3단계 - 개념, 논리, 물리

### 설계의 3단계 - 목표
- 1단계: 개념적 설계 (Conceptual Design)
    - 목표: 비즈니스의 아이디어나 요구 사항을 이해하고, 현실 세계의 정보들을 컴퓨터 세상의 언어로 번역하기 위한 밑그림
    - 질문: 우리가 다루어야 할 데이터는 무엇이며(Entity), 그 데이터들은 서로 어떤 관계(Relationship)를 맺고 있는가?
    - 산출물: ERD(Entity - Relationship Diagram): 사람의 눈으로 가장 이해하기 쉬운 형태의 설계도

- 2단계: 논리적 설계 (Logical Design)
    - 목표: 관계형 데이터베이스의 원리에 맞게 구체적인 구조로 다듬는 단계, 이때 특정 RDB에 종속되지 않음
    - 질문: 개념 모델의 각 요소들을 어떤 테이블 구조로 표현을 할 것인가? 데이터의 중복을 막고 관계를 명확히 하기 위해 어떤 규칙(정규화)를 적용해야하는가?
    - 산출물: 정규화된 테이블 스키마: 고객(고객 테이블), 상품(상품 테이블), 속성, 기본키, 외래키 관계

- 3단계: 물리적 설계 (Physical Design)
    - 목표: 논리적 설계에서 만든 테이블 스키마를 특정 RDBMS의 특성에 맞게 최적화
    - 질문: 각 컬럼에 어떤 데이터 타입을 할당할까? 인덱스를 어떻게 할까?
    - 산출물: SQL 스크립트, 테이블 정의서

### 설계의 3단계 - 용어 정리
- 구분
    - 저장 구조
    - 세부 항목
    - 데이터 단위

- 개념 모델
    - 엔티티
    - 속성
    - 인스턴스

- 논리 모델
    - 릴레이션
    - 속성
    - 튜플

- 물리 모델
    - 테이블
    - 열, 컬럼
    - 행
